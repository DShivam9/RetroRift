             RETRORIFT (RetroPlayHUB) — Interview Preparation Guide

Use this document to confidently explain your project in technical interviews.

================================================================================
 1. PROJECT OVERVIEW (The "Elevator Pitch")
================================================================================

"RetroRift is a full-stack retro game emulator hub I built from scratch using
React and Vite. It lets users browse a curated game library, play classic
console games (GBA, NES, SNES, NDS, Sega CD) directly in the browser using
EmulatorJS, manage save states, track achievements, and personalize their
profile — all wrapped in a polished cyberpunk/retro UI with custom animations
and effects."

Key talking points:
  • It's a COMPLETE application — not just a demo or tutorial project
  • 5 full pages, 35+ React components, custom design system
  • Real emulation — actually loads and runs ROM files in the browser
  • Build-time automation — wrote a Node.js scanner that auto-generates the
    game catalog from the file system
  • No CSS framework dependency — built with vanilla CSS using BEM methodology
    and CSS custom properties (design tokens)

================================================================================
 2. TECH STACK
================================================================================

  Frontend:
    • React 18 (functional components, hooks, lazy loading, Suspense)
    • Vite 7 (build tool, HMR, code splitting)
    • Vanilla CSS with BEM methodology (NOT Tailwind — deliberate choice)
    • CSS Custom Properties (design tokens for theming)
    • Lucide React (icon library)
    • Framer Motion (animations)
    • EmulatorJS (browser-based retro game emulator engine)

  Build Tooling:
    • Node.js scripts for ROM scanning and bulk downloading
    • ESM modules (import/export, not CommonJS)
    • npm lifecycle hooks (predev, prebuild) for automation

  Deployment:
    • GitHub Pages (static hosting)
    • CI/CD via GitHub Actions

  State Management:
    • React useState + useEffect (no Redux — deliberate simplicity)
    • localStorage for persistence (favorites, save states, play history)
    • React Context API for global settings (CRT filter, audio, scanlines)

 3. ARCHITECTURE & DESIGN DECISIONS
================================================================================

  3A. Component Architecture
  ─────────────────────────

  "I organized the project into a clean separation of concerns:"

    src/
    ├── pages/           → Full page components (5 pages)
    ├── components/      → Reusable UI components (35+)
    ├── components/effects/ → Visual effect components (CRT, LaserFlow, PixelSnow)
    ├── data/            → Auto-generated game catalog
    ├── lib/             → Utility libraries (emulator wrapper)
    ├── context/         → React Context for global state
    ├── hooks/           → Custom hooks (retro sounds)
    └── styles/          → Design system (tokens, animations, BEM components)

  3B. Routing
  ───────────

  "I implemented hash-based client-side routing without React Router. The App
  component reads window.location.hash, uses a pageKey state to force
  re-renders on navigation, and handles browser back/forward via the popstate
  event. This keeps the bundle smaller and gives me full control over
  transitions."

  Why not React Router?
    → It's a static site on GitHub Pages — hash routing is simpler
    → No server-side rendering needed
    → Reduces bundle size
    → Full control over page transition animations

  3C. State Persistence
  ─────────────────────

  "All user data is persisted in localStorage — favorites, save states, play
  history, and settings. I serialize/deserialize with JSON and wrap in
  try/catch to handle corrupt or missing data gracefully."

    • favorites       → array of game IDs
    • currentGame     → full game object (for page reload recovery)
    • playHistory     → last 20 played games with timestamps
    • saveSlots       → base64-encoded emulator save states per game
    • settings        → CRT filter, scanlines, audio prefs

  3D. CSS Architecture
  ────────────────────

  "I initially used Tailwind but deliberately migrated to vanilla CSS with BEM
  naming conventions. This gave me better performance (no unused utility
  classes), more readable markup, and full control over the design system."

    • Design tokens (CSS custom properties) for colors, spacing, typography
    • BEM methodology: .block__element--modifier
    • Component-scoped CSS files (GameCard.css, Navbar.css, etc.)
    • Consolidated from 4 navbar CSS files → 1 clean file

  3E. Build-Time Automation (ROM Scanner)
  ───────────────────────────────────────

  "I built a Node.js script that runs at build time (via npm lifecycle hooks)
  to automatically generate the game catalog. It scans the public/roms/
  directory, detects console types from file extensions, matches thumbnails,
  merges rich metadata from a JSON file, and outputs a complete JavaScript
  module."

  This is a STRONG interview talking point because it shows:
    • Build automation thinking
    • File system operations in Node.js
    • Data pipeline design (scan → transform → generate)
    • Developer experience focus ("drop a ROM, run dev, done")

 4. KEY FEATURES TO DISCUSS
================================================================================

  4A. Browser-Based Game Emulation
  ────────────────────────────────
  "I wrapped the EmulatorJS library in a custom GBAEmulator class that handles
  ROM loading, save/load states, and cleanup. The emulator supports multiple
  console types (GBA, NES, SNES, NDS, Sega CD) — the system type is auto-
  detected from the ROM file extension."

  Technical depth:
    • ROM files are fetched as ArrayBuffers via fetch()
    • Save states are serialized to base64 and stored in localStorage
    • Proper cleanup in useEffect return to prevent memory leaks
    • Canvas element for rendering, with CRT/scanline overlay effects

  4B. Save State Management
  ─────────────────────────
  "Users can save/load game progress at any point. Save states are stored in
  localStorage with per-game isolation. I also built an export/import system
  so users can back up their saves as files."

  4C. Global Search (Ctrl+K)
  ──────────────────────────
  "I implemented a Spotlight-style search modal (Ctrl+K shortcut) that filters
  games by title, console, genre, and developer in real-time. It supports full
  keyboard navigation — arrow keys to move, Enter to select."

  4D. Achievement System
  ──────────────────────
  "I created an unlockable achievement badge system — 8 achievements tracked
  client-side that trigger based on user actions (first game played, 10 games
  played, etc.)."

  4E. Lazy Loading & Code Splitting
  ─────────────────────────────────
  "All 5 pages are lazy-loaded using React.lazy() and Suspense. This means the
  initial bundle only loads the HomePage code — other pages are fetched on
  demand, reducing the initial load time significantly."

  4F. Responsive Design
  ─────────────────────
  "The app is fully responsive. The navbar collapses into a hamburger menu on
  mobile, game grids adjust their column count, and the player page reflows
  to a single-column layout on smaller screens."

 5. CHALLENGES & HOW I SOLVED THEM
================================================================================

  Challenge 1: Emulator Memory Leaks
  ───────────────────────────────────
  "The emulator creates DOM elements and loads large ROM ArrayBuffers.
  If you navigate away without cleanup, it leaks memory. I solved this by
  implementing a proper destroy() method and calling it in the useEffect
  cleanup function."

  Challenge 2: CSS Scalability
  ────────────────────────────
  "The project started with Tailwind, but as it grew to 35+ components, the
  utility-class approach made JSX unreadable. I refactored to BEM CSS with
  design tokens — more maintainable at scale and eliminated unused CSS."

  Challenge 3: Build-Time Game Catalog
  ────────────────────────────────────
  "Initially, adding a new game meant editing 3 different files manually. I
  automated this entirely with a Node.js scanner script that runs on npm
  lifecycle hooks (predev/prebuild), so now adding a game is just dropping
  a file into a folder."

  Challenge 4: Multi-System Emulator Support
  ──────────────────────────────────────────
  "Different consoles need different emulator cores. I abstracted this behind
  a unified GBAEmulator class that accepts a system parameter and configures
  the correct EmulatorJS core automatically."

  Challenge 5: Save State Persistence
  ────────────────────────────────────
  "Save states are large base64 strings (can be several MB). I implemented
  per-game isolation with indexed slot numbers, and added export/import so
  users don't lose progress if they clear browser data."

 6. COMMON INTERVIEW QUESTIONS & ANSWERS
================================================================================

  Q: "Why did you choose React for this project?"
  A: "React's component model is perfect for a UI-heavy application like this.
     I needed reusable components (GameCard, Toast, SearchModal), efficient
     re-rendering for the emulator canvas, and a strong ecosystem. I used
     React 18's features including Suspense for code splitting and lazy
     loading."

  Q: "Why Vite instead of Create React App?"
  A: "Vite is significantly faster — instant HMR vs CRA's webpack rebuild.
     It also has better code splitting out of the box, native ESM support,
     and a smaller output bundle. CRA is also deprecated."

  Q: "Why not use a state management library like Redux?"
  A: "For this project's scale, useState + useEffect + Context is sufficient.
     Redux would add complexity without benefit. The state is relatively
     simple — favorites (array), current game (object), settings (object).
     localStorage handles persistence. If the app needed a backend with
     real-time sync, I'd consider Redux or Zustand."

  Q: "How does the emulator work?"
  A: "I use EmulatorJS, which compiles retro console emulator cores to
     WebAssembly. My wrapper class handles ROM loading (fetching the file as
     an ArrayBuffer), initializing the correct emulator core based on the
     console type, and provides methods for save/load states. The emulator
     renders to an HTML5 Canvas element."

  Q: "How do you handle errors?"
  A: "ROM loading failures are caught in try/catch and displayed in the UI
     with a friendly error message. Toast notifications provide feedback for
     user actions. The app uses Error Boundaries for component-level errors,
     and all localStorage operations are wrapped in try/catch to handle
     corrupt data."

  Q: "What would you improve if you had more time?"
  A: "Three things:
     1. Backend integration — Firebase or Supabase for user auth and cloud
        save sync, so users can access saves across devices
     2. PWA support — service worker for offline play, installable as an app
     3. Performance — WebP/AVIF image optimization, more aggressive code
        splitting, and a virtual scrolling library for large game libraries"

  Q: "How did you handle routing?"
  A: "I implemented hash-based routing manually. The App component reads
     window.location.hash to determine the current page, uses
     history.pushState for navigation, and listens to the popstate event for
     browser back/forward. Each page is lazy-loaded with React.lazy()."

  Q: "Tell me about a technical decision you made."
  A: "Migrating from Tailwind to BEM CSS. Initially I used Tailwind for rapid
     prototyping, but at 35+ components the JSX became unreadable — 10-15
     utility classes per element. I refactored to BEM with CSS custom
     properties. This improved readability, eliminated unused CSS, and gave
     me a proper design system. The tradeoff was more upfront CSS writing,
     but the maintainability gain was worth it."

  Q: "How do you ensure code quality?"
  A: "ESLint for code style, component-scoped CSS files to prevent style
     leaks, React.memo for performance-critical components like GameCard,
     proper useEffect cleanup to prevent memory leaks, and meaningful
     component naming with JSDoc comments."

  Q: "What's the most complex component?"
  A: "PlayerPage — it orchestrates the emulator lifecycle (init, load ROM,
     save/load states, cleanup), manages session timing, renders game
     metadata, handles similar game recommendations, and integrates with
     the favorites system. It has to carefully manage side effects to prevent
     memory leaks when users navigate away mid-game."
